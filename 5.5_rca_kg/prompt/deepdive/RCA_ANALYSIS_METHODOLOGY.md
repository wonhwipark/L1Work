# RCA 원인분석 방법론 — L1SW 출력에서 root_cause 도출하기

작성일: 2026-06-23 KST
적용 대상: 5.5 RCA Knowledge Graph 분석 단계 (E2E 프롬프트 P4, scripts/README §8)
선행 문서: `L1SW_AND_FLOW_20260620.md`(L1SW가 멈추는 6개 공백), `RCA_FEEDBACK_LOOP_DESIGN_20260622_KST.md`(워크플로)

---

## 0. 이 문서가 푸는 문제

기존 자료는 **워크플로**(root_cause를 어디에 쓰는지, confidence를 어떻게 표기하는지,
unresolved를 언제 분리하는지)는 정의했지만, **"L1SW가 뱉은 `_l1sw.txt`에서 실제로 어떻게
원인을 찾아내는가"** 하는 추론 방법은 비어 있었다. 이 문서가 그 공백을 채운다.

skills_seed의 issue_type별 체크리스트는 "무엇을 볼지"(evidence 목록)는 주지만 "어떤 순서로
어떻게 판단할지"(추론 절차)는 주지 않는다. 이 문서는 **issue_type에 무관한 공통 추론
엔진**이고, skills_seed는 그 엔진에 꽂는 issue_type별 단서 사전이다.

핵심 목표 두 가지:
- **성공률**: 틀린 root_cause를 confirmed로 올리는 일을 줄인다 (false positive 억제).
- **효율**: 700MB 로그를 무작정 읽지 않고, 가설을 먼저 세워 **확인할 줄만 골라 본다**.

---

## 1. 대원칙 — 가설 주도(hypothesis-driven), 증거 역추적

원인분석은 "로그를 처음부터 읽는" 작업이 아니다. 그렇게 하면 700MB는 물론 5MB
`_l1sw.txt`도 다 못 읽는다. 대신:

```
증상(symptom) → 가설 후보 N개 → 각 가설이 참이라면 로그에 있어야 할 증거 →
그 증거를 cptime 구간에서 확인/반증 → 남은 가설이 root_cause
```

이것은 의료 감별진단과 같다. "열이 난다"에서 모든 검사를 하지 않고, 가능한 병 몇 개를
세운 뒤 그것을 가르는 검사만 한다. **로그 grep은 검사이고, signature는 증상이다.**

> 왜 효율적인가: L1SW가 이미 모듈·시간으로 1차 필터한 `_l1sw.txt`에서, 5.5는 **가설을
> 가르는 signature만** 추가로 grep한다. 전체 정독이 아니라 표적 확인이다.

## 2. 7단계 분석 절차 (issue_type 공통)

### S1. 증상 고정 (Symptom anchoring)
- `_l1sw.txt`에서 **실패를 단정하는 signature**(role: `failure_event`)의 **첫 등장
  cptime**을 찾는다. 이게 분석의 원점(anchor)이다.
- 예: RACH면 `RACH_PHY_TIMER_EXPIRY`/`RACH_RAR_TIMEOUT`, SCG면 `SCG_RLF`/`SCG_T310`.
- failure_event가 여러 개면 **가장 이른 것**을 원점으로. 뒤의 것은 연쇄 결과일 수 있다.

### S2. 시간창 설정 (Window around anchor)
- anchor cptime 기준 **앞으로** 충분히(예: 진입 이벤트까지), **뒤로** 약간(fallback/해제
  동작까지) 구간을 잡는다. 이 구간이 `cptime_range` 후보가 된다.
- 앞을 보는 이유: **원인은 항상 결과보다 먼저 cptime에 찍힌다.** 뒤를 보는 이유: 진짜
  실패인지(복구 못 함) 일시적 경고인지(곧 성공) 구분.

### S3. 정상 경로 대조 (Happy-path delta)
- 해당 issue_type의 **정상 시퀀스**를 기준선으로 둔다.
  - RACH: `Msg1→Msg2(RAR)→Msg3→Msg4(contention resolution)`
  - SCG:  `measReport→SCG-Config→randomAccess→SCG 활성`
- anchor 구간의 실제 시퀀스를 정상 시퀀스와 겹쳐, **어느 단계에서 처음 이탈했는가**를
  특정한다. **최초 이탈 지점(first divergence)이 root_cause에 가장 가깝다.**
- 예: `Msg1→Msg2→Msg3→(타임아웃)`이면 Msg3 이후 응답 단계 이탈 → Msg4/contention 또는
  상위 grant 문제로 가설이 좁혀진다.

### S4. 가설 후보 나열 (Differential)
- skills_seed의 **Root Cause Categories 표**를 가설 풀(pool)로 사용한다.
- S3의 최초 이탈 지점과 **양립 가능한 category만** 남긴다. (이탈이 Msg3 전이면
  `scheduler_grant_missing`은 후보에서 빠진다 — grant는 Msg3 이후 단계.)
- 보통 2~4개로 좁혀진다. 1개로 즉단하지 말 것(확증 편향).

### S5. 가설 가르기 (Discriminating evidence)
- 각 가설을 **참/거짓으로 가르는 단 하나의 결정적 단서**를 정하고 그것만 확인한다.
  | 가설 | 가르는 단서(있으면 ↑, 없으면 ↓) |
  |---|---|
  | radio_access_timeout | 동일 cell 타 UE는 RACH 성공? → 성공이면 단일 UE 무선문제 ↑ |
  | preamble_collision | 재시도 시 동일 preamble 반복? |
  | scheduler_grant_missing | Msg3 직전 UL grant 로그 존재? |
  | config_parameter_mismatch | timeout 값이 규격 대비 비정상으로 짧음? |
  | api_contract_mismatch | REQ/CNF 파라미터 불일치 로그? |
- **하나의 grep으로 두 가설을 동시에 가를 수 있으면 그것부터** 한다(효율).

### S6. 단일 원인 vs 환경 요인 분리 (Scope test)
- "이 UE만의 문제"인지 "cell/환경 문제"인지 가른다.
  - 동일 cell 내 **다른 UE가 같은 시간대에 성공**했나? → 성공이면 단말 측, 전부 실패면
    cell/네트워크/RF 측.
- **환경/RF 측으로 판정되면 → 담당영역 밖 → `unresolved` + handoff** (원인 미상이 아니라
  "내 도메인 아님"). 이 분기를 S6에서 명시적으로 건다.

### S7. 인과사슬 확정 + confidence 산정
- 남은 가설을 **인과사슬**로 적는다(증상 아님, 원인→결과 연결):
  `UL grant 누락 → Msg3 미전송 → RAR 재시도 → PHY_TIMER_EXPIRY → RACH 실패`
- 사슬의 **모든 화살표가 로그 증거로 뒷받침되면** confidence ↑.
  끊긴 화살표(추정으로 메운 곳)가 있으면 그 수만큼 confidence ↓.

## 3. confidence 산정 기준 (자의적 표기 금지)

`root_cause.confidence`는 느낌이 아니라 **증거 충족도**로 정한다.

| confidence | 조건 |
|---|---|
| `confirmed` | 인과사슬 전 구간 로그 증거 존재 + 가설 가르는 단서 명확 + 재현/타UE 대조 완료 |
| `high` | 인과사슬 대부분 증거 존재, 화살표 1개만 합리적 추정 |
| `medium` | 최초 이탈 지점은 특정됐으나 그 위 원인은 2개 가설이 남음 |
| `low` | 증상은 분류됐으나 최초 이탈 지점 위로 단서 부족 (담당영역 안, 추가 분석 필요) |

> **중요한 구분**(FEEDBACK_LOOP §11 재확인):
> - 담당영역 안 + 원인 불확실 = `status: analyzed` + `confidence: low`
> - 담당영역 밖 = `status: unresolved` + handoff (confidence 표기 안 함)
> 둘을 섞지 말 것.

## 4. 효율 규칙 (큰 로그에서 시간 낭비 안 하기)

1. **anchor부터, 바깥으로**: 파일 처음부터 읽지 말고 failure_event cptime을 먼저 찾아
   거기서 양방향으로 확장한다.
2. **가설이 부르는 grep만**: S5의 "가르는 단서"에 해당하는 signature만 추가 검색한다.
   전체 signature를 다 세지 않는다.
3. **generic은 위치 좁힐 때만**: `RACH`/`SCG`/`RLC`/`HARQ`(noise_risk:high)는 원인 판정에
   쓰지 말고 "이 구간이 맞나" 문맥 확인용으로만. fingerprint·root_cause 근거로 금지.
4. **cptime 단조성 활용**: 원인은 결과보다 cptime이 작다. 큰 cptime 줄을 원인 후보로
   올리지 말 것.
5. **3회 규칙**: 같은 가설을 가르려고 3번 넘게 grep했는데 결론이 안 나면, 그 가설은
   `low`로 두고 다음으로 넘어간다(무한 정독 방지).

## 5. 실패를 부르는 안티패턴 (이걸 하면 성공률이 떨어진다)

- **증상을 원인으로 적기**: "PHY_TIMER_EXPIRY 발생"은 증상이다. 원인은 "왜 타이머가
  만료됐는가"다. root_cause.summary에 signature 이름만 적으면 실패.
- **첫 가설 확증**: 처음 떠오른 category의 증거만 찾고 반증을 안 본다. S4에서 최소 2개를
  강제로 세우는 이유.
- **연쇄결과를 원점으로**: 늦게 찍힌 failure_event(예: 연결 해제)를 anchor로 잡으면 진짜
  최초 이탈을 놓친다. S1에서 "가장 이른 failure_event".
- **wall-clock으로 순서 판단**: 시험마다 시계가 다르다. 순서는 **항상 cptime**.
- **담당영역 밖인데 low로 우기기**: RF/환경 문제를 `confidence:low`로 들고 있으면 영원히
  안 풀린다. S6에서 unresolved로 빠르게 넘긴다.
- **generic signature로 fingerprint**: false 매칭 양산. §4-3 위반.
- **보이는 범위에서 원인 확정 (가로등 밑에서 열쇠 찾기)**: `_l1sw.txt`에 떠온 signature 중에서만
  원인을 고르면, 진짜 원인 모듈이 필터에서 빠졌을 때 "보이는 것 중 best"를 원인으로 단정하게
  된다. 최초 이탈 지점의 **원인 줄이 안 보이면** confidence를 올리지 말 것. → §5.5 미지 모듈 가드.

## 5.5 미지 모듈 가드 (most important — 오분석을 막는 핵심 안전장치)

### 왜 필요한가
5.5는 `_l1sw.txt`에 **있는 것만** 본다(원본 `.sdm`은 안 읽는다). 그래서 진짜 원인이 찍힌
모듈이 L1SW manifest/필터에서 빠지면, 그 줄은 `_l1sw.txt`에 아예 없다. 이때 분석기는
"원인 없음"을 깨닫는 게 아니라, **보이는 signature 중 그럴듯한 것을 원인으로 골라버린다.**
더 위험한 건 보이는 범위 안에서는 인과사슬이 매끄럽게 이어져 **confidence가 거짓으로 높아진다**.

### 어떻게 탐지하나 (S3에 직접 연결)
S3에서 정상 시퀀스와 대조해 "최초 이탈 지점"을 찾았는데, **그 이탈을 일으킨 원인 줄이
`_l1sw.txt`에 없으면** = 미지 모듈 누락 신호다. 구체적으로:
- 최초 이탈 지점은 특정됐다(예: Msg3 이후 응답 단계에서 끊김).
- 그런데 그 위(원인 쪽) cptime 구간에 **설명할 signature가 비어 있다**(공백/침묵).
- 이 "이탈 지점 위의 공백"이 바로 경보다. 공백을 추정으로 메우지 말 것.

### 무엇을 해야 하나
1. **confidence를 올리지 않는다.** `status: analyzed` + `confidence: low` 로 둔다.
   (담당영역 밖이 아니므로 `unresolved`가 아니다 — 아래 구분 표 참조.)
2. **root_cause.summary에 "원인 줄 부재"를 명시한다.** 예: "최초 이탈은 Msg3 후 응답 단계.
   원인 구간(cptime X~Y) 로그가 `_l1sw.txt`에 없어 원인 모듈 미상 — L1SW 필터 보강 필요."
3. **L1SW 필터를 보강해 `_l1sw.txt`를 다시 만든다**(RUNBOOK STEP F 막힘 분기):
   - 의심 시간창에서 **모듈 필터를 풀고**(또는 넓혀) 재추출 → 어떤 모듈이 그 시간대에
     활동했는지 본다.
   - 새로 드러난 모듈을 manifest/keywords에 후보로 추가 → 다시 P3 → P4.
   (추가 추출의 고지·횟수·종결 규칙은 아래 "추가 추출 프로토콜" 참조.)

### "미지 모듈" vs "담당영역 밖"(unresolved) 구분 — 섞지 말 것
| 상황 | 무엇이 부족한가 | 처리 |
|---|---|---|
| 미지 모듈 누락 | 원인이 담당 안에 있을 수 있으나 **로그 가시성 부족** | `analyzed`+`low` + 필터 보강 후 재분석 |
| 담당영역 밖 | 원인이 RF/환경 등 **내 도메인 밖** | `unresolved` + handoff |

둘 다 confidence를 억지로 올리지 않는다는 점은 같지만, **조치가 다르다**. 미지 모듈은
"내가 더 떠와서 다시 본다", 담당영역 밖은 "이관한다".

### 추가 추출 프로토콜 (누가·언제·몇 번 — 임의 재추출 금지)
미지 모듈이 감지되면 아래 절차를 **반드시 이 순서로** 따른다. 분석기(Claude Code)가
혼자 로그를 다시 떠서는 안 된다 — L1SW 실행은 사람 권한이다.

1. **명시 고지 (자동 재추출 금지)**: 분석기는 "추가 추출이 필요하다"를 먼저 **명확히
   고지**하고 멈춘다. 고지에 다음을 포함한다:
   - 왜 필요한가: 어느 cptime 구간 / 어느 이탈 지점 위가 비었는가
   - 무엇을 어떻게 넓힐까: 제안하는 시간창(이탈 직전으로 확장)과 모듈 필터(풀기/추가할 모듈)
   - **L1SW 재실행 명령(안)**을 제시하되 직접 실행하지 않는다.

1.5. **키워드 보강 (추출 전 필수 — SSOT 준수)**: 추가 추출은 결국 "필터를 넓히는 것"이고
   필터의 출처는 `keywords.yaml`(SSOT)이다. 따라서 새 키워드는 **반드시 `keywords.yaml`에
   해당 issue_type 아래 `status: candidate` 로 먼저 추가**한 뒤 추출에 쓴다. ps1/manifest에
   직접 넣지 않는다(드리프트 방지). confirmed 로는 절대 바로 올리지 않는다.

   누가 키워드를 정할지는 **사람에게 묻는다**. 이때 "A/B/C 중 고르세요" 식으로 던지지 말고,
   처음 보는 사람도 답할 수 있게 **각 선택지의 뜻·입력 형식·예시까지 풀어서** 질문한다.
   질문 템플릿(분석기가 이렇게 묻는다):

   ```
   원인으로 의심되는 로그가 현재 _l1sw.txt 에 없어, 필터를 넓혀 다시 추출해야 합니다.
   어떻게 키워드를 보강할지 골라 주세요. (처음이셔도 아래 설명대로 하시면 됩니다.)

   [A] 제가 찾은 후보 모듈을 추가             ← 추천. 가장 쉬움
       P3가 본 "이 시간대 등장 모듈" 중 원인일 가능성이 있는 것을 제가 골라뒀습니다:
         · <모듈명1> (이유: 이탈 직전 cptime에 활동)
         · <모듈명2> (이유: 정상 케이스엔 없는데 이번에만 등장)
       → 이 중 추가할 번호를 알려 주세요 (예: "1번", "1,2번", "전부").

   [B] 제가 직접 키워드를 입력               ← 도메인 지식이 있을 때
       찾고 싶은 로그의 키워드/모듈명을 적어 주세요.
       형식: 쉼표로 구분. 예: "ROHC, PDCP_DISCARD, t-Reordering"
       (정규식 몰라도 됩니다. 평범한 단어로 적으면 제가 keywords.yaml 형식에 맞춥니다.)

   [C] 키워드는 그대로, 모듈 필터만 풀고 넓게 추출   ← 뭘 넣을지 모를 때
       의심 시간창에서 모듈 제한 없이 통째로 떠서, 어떤 모듈이 있는지 먼저 봅니다.
       → "C" 라고만 답하시면 됩니다. (가장 안전하지만 결과가 큼)

   [D] 추가 추출 안 함 — 지금 미결로 종료              ← 더 파지 않기로 할 때
       시간이 없거나, 지금 로그로는 어렵다고 판단되면 추가 추출을 멈추고 미결로 닫습니다.
       → "D" 라고 답하시면, 이 case를 status: analyzed + confidence: low 로 남기고
         summary에 "추가 추출 미수행 — 사용자 판단으로 미결 종료"라고 기록합니다.
         (나중에 다시 열어 추출을 이어갈 수 있습니다. 정보는 버리지 않습니다.)

   [E] 제가 원인을 찾았습니다 — 정보 넣고 마무리          ← 직접 원인을 파악했을 때
       분석 중에 원인을 아셨다면, 딱 두 가지만 알려주세요. 나머지는 제가 알아서 정리합니다.
         (1) 원인이 뭔가요?  예: "PDCP reordering 타이머가 너무 짧아 패킷 폐기 → RLC 재전송 폭증"
         (2) 그 근거 로그가 있나요?
              · 있음 → 그 줄이나 cptime 을 알려주세요 (예: "cptime 12345의 PDCP_DISCARD 로그")
              · 없음(경험/추정) → "추정" 이라고만 하세요
       → 이것만 주시면 제가: case의 root_cause 를 채우고, 관련 모듈을 keywords 후보로 넣고,
         검토 단계로 넘긴 뒤 마무리합니다. status/저장위치/형식은 신경 안 쓰셔도 됩니다.
       ※ 혹시 "지금 보던 것과 다른 모듈의 문제"를 발견하신 거면, 그냥 그렇게 한 줄 적어 주세요.
          (예: "이건 RACH 말고 SCG 쪽 문제 같음") 제가 이번 case 를 오염시키지 않고 별도 발견
          메모로 따로 남깁니다.

   잘 모르겠으면 [C] 를 권합니다. 먼저 넓게 본 뒤 다음 번에 좁히면 됩니다.
   지금 멈추고 싶으면 [D], 원인을 이미 아시면 [E] 입니다. 틀린 선택은 없습니다.
   ```

   - 답을 받으면 선택된 키워드를 `keywords.yaml`에 `candidate`로 추가하고(추가한 ID 목록을
     사람에게 보여줌), 그 candidate 를 포함해 prefilter 를 구성한다.
   - [C]는 키워드 추가 없이 모듈 필터만 푼다(넓은 1패스). 결과에서 새 모듈이 보이면 그때
     [A]로 돌아와 candidate 추가.
   - **[D]는 추가 추출을 하지 않고 미결로 종료**한다. case를 `status: analyzed` +
     `confidence: low` 로 남기고 summary에 "사용자 판단으로 미결 종료"를 명시한다.
     keywords는 보강하지 않는다. 이는 **사람이 능동적으로 고른 종결**로, 2회 추출 소진 후의
     자동 "원인 미상"(3번 항목)과 구분된다 — 둘 다 미결이지만 D는 "지금 안 함", 자동은
     "다 해봤는데 안 됨"이다. 어느 쪽이든 나중에 case를 다시 열어 추출을 재개할 수 있다.
   - **[E]는 사람이 직접 규명한 원인을 KG에 흡수하고 종료**한다. 사람은 (1)원인 (2)근거 로그
     유무, 두 가지만 답하고 나머지는 **시스템이 자동 처리**한다(사용자 편의 원칙 — status·저장
     위치·형식을 사람에게 묻지 않는다):
     · root_cause.summary = 사람이 준 인과사슬 문장. root_cause.evidence = 근거 로그(있으면).
     · confidence 자동 산정: **근거 로그를 제시하면 high, "추정"이면 medium**(증거 없이 confirmed
       금지). 절대 사람 입력만으로 자동 confirmed 로 올리지 않는다.
     · status = `reviewed` 로 두고 P5 검토를 자동으로 거치게 한다(사람도 실수하므로 점검 1회).
       사람 입장에선 "원인 적었더니 알아서 검토로 넘어감" — 추가 질문 없음.
     · 사람이 짚은 모듈/패턴은 `keywords.yaml`에 `candidate`로 추가하되 note에
       **"source: 분석자 직접 규명 (case_id)"**를 기록(출처 추적). 실로그 근거가 함께 있으면
       P6에서 confirmed 승격 대상.
     · **별개 모듈 발견 시(흐름 끊지 않기)**: 사람이 "다른 모듈 문제"라고 적으면, 이번 case에
       섞지 말고 `cases/findings/` 또는 case의 `related_findings` 메모로 **가볍게 따로** 남긴다
       (오염 방지). 그 자리서 새 case 를 강제 생성하지 않는다 — 나중에 별도 P3~P4 로 제대로 분석.
       사람은 "이것도 문제 있더라" 한 줄만 남기면 끝.
2. **사람이 보강 추출**: 사람이 그 제안대로 L1SW 필터를 보강해 `_l1sw.txt`를 재생성한다
   (RUNBOOK STEP E 반복). 새 `_l1sw.txt` 로 P3 → P4 재실행.
3. **재시도 상한 = 누적 2회**: 추가 추출은 **최대 2회까지**만 한다.
   - 1회차 보강 후에도 원인 줄이 안 나오면 → "재시도"(2회차 추출 제안, 다른 축으로:
     1회차에 시간창을 넓혔으면 2회차엔 모듈 필터를 풀고 등).
   - **2회차(누적)에도 이탈 지점 위가 공백이면 → "원인 미상"으로 확정**한다.
     `status: analyzed` + `confidence: low` 고정 + summary에 "추가 추출 2회에도 원인 구간
     로그 부재 — 이 로그 범위로는 원인 규명 불가" 명시. 더 이상 추출 반복하지 않는다.
4. **추출 이력 기록**: 각 추출 시도의 (시간창, 모듈 필터, 추가한 candidate 키워드, 결과)를
   case의 분석 메모 또는 review_log에 남겨, 같은 범위를 무한 반복하지 않게 한다.

> 핵심: ① 고지하고 멈춘다(자동 X) ②키워드 보강/종결 선택(A 후보모듈 / B 직접입력 / C 모듈필터만
> 풀기 / D 미결 종료 / E 사람이 원인 규명·자동 반영) ③ 사람이 보강 실행 ④ 2회까지 ⑤ 원인 미상 확정.
> 사용자 편의: 사람은 최소만 답하고(E면 원인+근거 2개) status·저장위치·형식은 시스템이 자동 처리.
> "원인 미상"은 실패가 아니라 **정직한 종결**이다 — 억지 결론보다 KG 신뢰도에 낫다.

### 예방 습관 (S2 단계에서 미리)
처음부터 `-Modules`로 좁히지 말고, **의심 시간창만 잡아 모듈 필터 없이 한 번** `_l1sw.txt`를
떠서 "이 시간대에 어떤 모듈이 활동했나"를 본 뒤 좁힌다. 미지 모듈은 이 넓은 1패스에서
드러난다. (P3 보고에 "이 시간창 등장 모듈 전체 목록"을 함께 출력하면 사람이 눈치챈다.)

## 6. 분석 결과를 case에 매핑 (schema v2 연결)

| 분석 산출 | case 필드 |
|---|---|
| S1 anchor signature | `fingerprint.signature_set`(ID), `log_patterns[].signature` |
| S2 시간창 | `log_patterns[].cptime_range` |
| S3 최초 이탈 시퀀스 | `fingerprint.sequence` (cptime 정렬, ID 표기, 절대시각 금지) |
| S4~S5 채택 category | `root_cause.category` (taxonomy active 목록) |
| S7 인과사슬 | `root_cause.summary` (사슬 문장) |
| S7 confidence | `root_cause.confidence` (§3 기준) |
| S5 가르는 단서 줄 | `root_cause.evidence` → `log_patterns[].pattern_id` |
| S6 환경 판정 | `unresolved/` 분리 + `handoff` 블록 |

> signature는 **반드시 keywords.yaml ID**(`RACH_MSG1` 등, alias 금지). 2026-06-23 정합성
> 통일과 일치.

## 7. 한 줄 요약

```
증상 anchor 고정 → 시간창 → 정상경로 대조로 최초 이탈 특정 → 가설 2~4개 →
가르는 단서만 grep → 단말/환경 분리(환경이면 handoff) → 인과사슬+confidence
원칙: 가설이 부르는 줄만 본다. 증상이 아니라 원인을 적는다. 순서는 cptime.
가드: 최초 이탈 지점의 원인 줄이 _l1sw.txt에 없으면 = 미지 모듈 신호.
      보이는 것 중에서 원인 고르지 말고 low로 두고 L1SW 필터 보강 후 재분석(§5.5).
```

이 절차는 issue_type과 무관하다. skills_seed의 issue_type별 단서 표를 §2의 S3~S5에
꽂아 쓰면 4종(rach/scg/tx/l2) 모두 동일 엔진으로 분석된다.
