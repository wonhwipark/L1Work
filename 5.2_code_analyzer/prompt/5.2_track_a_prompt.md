## 0. 이 문서의 역할

Track A는 CodeAnalyzer의 메인 실행 경로다.

```text
코드루트 + 블록명/API명
  → Phase 0 구조맵 확보
  → Phase 1 procedure 목록 + procedure_runtime_index 생성
  → Phase 2..N procedure별 call flow + 분리형 .puml MSC 생성
  → Phase F HLD성 md 마무리
```

Track B는 선택적 전처리다. Track B가 생성한 structure는 Track A Phase 0이 같은 canonical path에서 자동으로 찾는다.

## 1. v0.2 고정 규칙

### 1.1 output canonical path

모든 산출물은 아래 상대경로에 저장한다.

```text
output/5.2/<slug>/
```

Track A는 아래 경로를 사용하지 않는다.

```text
%USERPROFILE%\artifacts\code_analyzer\<slug>\
artifacts/5.2/<slug>/                            # v0.2 경로. v0.3에서 output/5.2/로 이동
artifacts\code_analyzer\<slug>\
output/5_2/<slug>/
```

### 1.2 slug 규칙

```text
1. 사용자가 target slug를 주면 최우선 사용
2. 없으면 block_or_api 이름에서 자동 생성
3. 소문자 변환
4. 공백, /, \, :, ., -, (, ), [, ], {, } 는 _ 로 변환
5. 연속 _ 는 하나로 축약
6. 앞뒤 _ 제거
```

### 1.3 extraction_mode enum

Track A가 직접 structure를 생성하거나 Track B 결과를 읽을 때 아래 enum만 사용한다.

```text
extraction_mode: git | bash | powershell | internal_search | mixed
```

### 1.4 progress cursor

모든 응답 마지막은 token progress cursor를 사용한다.

```text
[진행: PHASE0_DONE → 다음: PHASE1_PROCEDURE_DISCOVERY]
```

산문형 진행줄은 금지한다.

### 1.5 runtime read budget

v0.2의 토큰 절감 핵심은 reference prompt 분리뿐 아니라 런타임 반복 read 차단이다.

```text
1. Phase 2..N은 analysis_progress.md의 procedure_runtime_index에서 해당 procedure slice만 읽는다.
2. Phase 2..N은 structure_focused.json 전체를 매번 다시 읽지 않는다.
3. structure read는 procedure_runtime_index가 없거나 INDEX_INCOMPLETE일 때만 fallback으로 허용한다.
4. 전역 누적 call edge 본문 블록은 만들지 않는다.
5. procedure별 call edge는 각 procedure 섹션 내부에만 기록한다.
6. 전역에는 필요 시 edge id 목록만 남긴다.
7. MSC는 HLD md에 inline PlantUML로 넣지 않고 별도 `.puml` 파일로 저장한다.
```

## 4. Phase 0 — 구조맵 확보 + Stage 계획

### 실행 지시

```text
code-analyzer 스킬로 <코드루트경로> 를 분석할 준비를 해줘.

[입력]
- 코드 루트: <코드루트경로>
- 진입 모드: <블록명 또는 API명>
- target slug: <선택, 비우면 자동 생성>
- 분석 확장자: .c .cpp .h .hpp
- 제외 폴더: test/ third_party/ build/
- structure.json: <경로 또는 자동>

[Phase 0 조건]
1. slug를 확정한다.
2. canonical path를 확정한다: output/5.2/<slug>/
3. structure.json을 확보한다.
   - 경로가 명시되면 해당 파일을 사용하되 canonical path에 복사하지 말고 참조 경로를 기록한다.
   - 자동 탐색이면 아래 우선순위를 따른다.
     a. output/5.2/<slug>/structure_*_focused.json 최신 파일
     b. output/5.2/<slug>/structure_*.json 중 파일 크기 300KB 이하인 최신 파일
     c. 300KB 초과 full structure는 자동으로 열지 않는다. 사용자가 명시한 경우에만 사용한다.
   - 없으면 focused extraction으로 structure_<ts>_focused.json을 생성한다.
4. structure.json의 meta.root가 입력 코드루트와 다르면 경고하고 멈춘다.
5. structure.json의 meta.slug가 현재 slug와 다르면 경고하고 멈춘다.
6. 깊은 body 분석은 하지 않는다.
7. analysis_progress.md를 output/5.2/<slug>/ 아래 생성/갱신한다.
8. NEXT_STEP_5.2.md를 갱신한다.
9. 진행줄을 출력한다.
```

### Phase 0 직접 추출 제한

```text
- source body 전체 read 금지
- 파일 목록, LOC, 후보 함수명, 후보 line만 추출
- grep/검색 결과는 패턴별 최대 200줄까지만 1차 수집
- block/API 관련 후보 파일을 먼저 좁힌 뒤 1-hop, 2-hop만 확장
- structure.json이 300KB를 넘으면 summary mode
- 1MB를 넘으면 focused structure만 생성
```

### Phase 0 완료 진행줄

```text
[진행: PHASE0_DONE → 다음: PHASE1_PROCEDURE_DISCOVERY]
```

## 5. Phase 1 — procedure 목록 + procedure_runtime_index 도출

### 실행 지시

```text
output/5.2/<slug>/analysis_progress.md 와 선택된 structure json을 읽고 Phase 1을 수행해줘.
```

### 모드 A — 블록 기준

```text
1. <블록명>에 해당하는 handler 파일/함수 군을 식별한다.
2. 그 블록이 수신하는 외부 인터페이스를 열거한다.
   - IPC REQ
   - 메시지 handler
   - 공개 API
   - dispatch entry
3. 이 수신 인터페이스 목록을 procedure 후보로 만든다.
4. procedure 후보를 번호로 제시하고 멈춘다.
5. 사용자가 전부/일부를 한 번 선택하게 한다.
```

### 모드 B — API 기준

```text
1. <API명>을 entry point로 고정한다.
2. 단일 procedure로 확정한다.
3. 결과 위치를 1회 물어본다.
   - (가) 해당 블록 HLD .md에 합류
   - (나) 독립 hld_<proc>_<ts>.md 생성
4. 선택 후 Phase 2로 이동한다.
```

### Phase 1 필수 기록 — procedure_runtime_index

Phase 1은 structure json을 반복 read하지 않도록 `analysis_progress.md`에 작은 실행 인덱스를 반드시 기록한다.

```text
procedure_runtime_index:
  - procedure_slug: <procedure_slug>
    entry_point: <function/message/api>
    entry_file: <relative path>
    entry_line: <line 또는 UNKNOWN>
    related_files:
      - <relative path>
    req_site_ids:
      - <REQ_EDGE_ID 또는 UNKNOWN>
    cnf_handler_candidate_ids:
      - <CNF_HANDLER_ID>
    call_edge_ids:
      - <EDGE_ID>
    hld_section_anchor: <anchor>
    msc_file: output/5.2/<slug>/msc_<procedure_slug>_<YYYYMMDD_HHMM_KST>.puml
    index_status: READY | INDEX_INCOMPLETE
    rn:
      - <필요 시>
```

주의:

```text
- call_edge 본문 전체를 전역 누적 블록에 복사하지 않는다.
- 필요한 edge id와 관련 파일 포인터만 procedure_runtime_index에 기록한다.
- Phase 2..N이 해당 procedure slice만 읽고 시작할 수 있어야 한다.
```

### Phase 1 완료 진행줄

```text
[진행: PHASE1_DONE → 다음: PROC_NEXT:<procedure_slug>]
```

## 6. Phase 2..N — procedure별 call flow 추적 + 분리형 MSC

### 실행 지시

```text
output/5.2/<slug>/analysis_progress.md 를 읽고, 아직 DONE 아닌 procedure 하나만 분석해줘.
DONE procedure는 다시 분석하지 마.
```

### 조건

```text
1. progress_cursor와 next를 먼저 읽는다.
2. procedure_runtime_index에서 이번 procedure slice만 읽는다.
3. index_status가 READY이면 structure_focused.json 전체를 다시 읽지 않는다.
4. index_status가 INDEX_INCOMPLETE이거나 필요한 id가 없을 때만 structure json targeted fallback read를 허용한다.
5. 이번 procedure의 entry point부터 callee-side call flow를 추적한다.
6. HAL/PHY/IPC REQ boundary까지 api-callflow-analysis 스킬로 추출한다.
7. 미확인 flow는 [RN]으로 남긴다.
8. CNF handler는 단일 가정하지 않는다.
9. cnf_handler_candidate_ids 중 procedure와 매칭되는 handler를 targeted read로 확정한다.
10. 확정된 CNF handler는 Global CNF Carry Map에 SELECTED로 기록하고 재read하지 않는다.
11. 확정 불가 시 MSC에 [RN] CNF side pending 또는 [RN] CNF handler ambiguous 표시.
12. plantuml-msc 스킬로 procedure MSC 1개를 만든다.
13. MSC는 `msc_<procedure_slug>_<YYYYMMDD_HHMM_KST>.puml` 파일로 저장한다.
14. HLD .md에는 PlantUML inline block을 넣지 않고 `msc_ref` 링크와 동작 설명만 넣는다.
15. procedure별 call edge 본문은 해당 procedure 결과 섹션 내부에만 기록한다.
16. analysis_progress.md 갱신 후 멈춘다.
```

### MSC 구성

```text
entry
→ local validation / DB read
→ decision branch
→ internal callee
→ HAL/PHY/IPC REQ boundary
→ CNF/confirm boundary
→ completion / error branch
```

### Procedure 완료 진행줄

다음 procedure가 있으면:

```text
[진행: PROC_DONE:<procedure_slug> → 다음: PROC_NEXT:<next_procedure_slug>]
```

마지막 procedure면:

```text
[진행: PROC_DONE:<procedure_slug> → 다음: PHASEF_FINALIZE]
```

## 7. 이어하기 — resume

```text
output/5.2/<slug>/analysis_progress.md 를 읽고 이어서 분석해줘.

[조건]
1. progress_cursor를 먼저 확인한다.
2. canonical_path가 output/5.2/<slug>/ 인지 확인한다.
3. DONE procedure는 재분석하지 않는다.
4. Global CNF Carry Map에서 SELECTED handler는 재read하지 않는다.
5. next 필드가 가리키는 작업 하나만 수행한다.
6. next가 PROC_NEXT이면 procedure_runtime_index의 해당 slice만 읽는다.
7. structure json은 index_status가 INDEX_INCOMPLETE일 때만 fallback으로 읽는다.
8. 응답 마지막은 token progress cursor로 끝낸다.
```

## 8. Phase F — 블록 HLD성 md 마무리

```text
선택된 procedure가 모두 DONE이면 블록 HLD .md를 마무리해줘.

1. 블록 책임 1–2줄 작성
2. 외부 인터페이스 목록 작성
3. procedure 목록과 HLD 섹션 링크 작성
4. procedure별 `msc_ref`가 존재하고 `.puml` 파일이 저장되어 있는지 확인
5. HLD .md에 PlantUML MSC inline block을 넣지 않음
6. 열려있는 [RN] 목록 정리
7. hld_<block>_<YYYYMMDD_HHMM_KST>.md 최종본 저장
8. NEXT_STEP_5.2.md 갱신
```

완료 진행줄:

```text
[진행: BLOCK_HLD_DONE:<slug> → 다음: WAIT_NEW_BLOCK]
```

## 9. 검증 기준

```text
- artifact 경로가 output/5.2/<slug>/ 로 통일됨
- extraction_mode가 enum 중 하나임
- 진행줄이 token cursor 형식임
- 한 번에 하나의 procedure만 분석함
- Phase 1이 procedure_runtime_index를 생성함
- Phase 2..N이 structure json 전체를 반복 read하지 않음
- 전역 누적 call edge 본문 블록이 없음
- procedure별 call edge는 procedure 섹션 내부에만 있음
- MSC는 별도 .puml 파일로 저장됨
- HLD .md에는 .puml 링크와 산문 설명만 있음
- CNF handler 단일 가정 없음
- structure 자동 탐색은 focused 우선, full은 300KB 이하만 자동 사용
- raw AST 덤프 없음
- [RN] 없이 추측하지 않음
```
